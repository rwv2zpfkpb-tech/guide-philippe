### Tabellen

**`restaurants`** вҖ” KernentitГӨt
- `id` uuid PK
- `name` text (Pflicht)
- `google_place_id` text | null вҖ” fГјr Live-Daten von Google. `null` bei manuell erfassten Restaurants (s. вҖһManuelle Restaurant-Erfassung" unten) вҖ” kein Live-Lookup (Г–ffnungszeiten/Fotos) mГ¶glich, Admin-Tabelle zeigt вҖһNo place ID"-Badge
- `lat`, `lng` float | null
- `address` text | null (Migration `20260716000001_restaurant_address.sql`) вҖ” Adresse aus der Places-Autocomplete-Auswahl oder manuell erfasst; dient der Restaurant-Detailseite als Fallback, wenn kein `google_place_id` vorhanden ist oder der Live-Lookup fehlschlГӨgt
- `phone` text | null, `website` text | null (Migration `20260716000002_restaurant_contact_info.sql`) вҖ” wie `address`: standardmГӨГҹig live von Google Places bezogen (`getPlaceDetails()`), aber zusГӨtzlich persistiert als Fallback fГјr manuell erfasste Restaurants/fehlgeschlagenen Live-Lookup. Live-Wert hat immer Vorrang (`placeDetails?.phone || restaurant.phone`, s. Restaurant-Detailseite)

### Wie Auth-PrГјfung funktioniert

In Server Actions immer `requireAuth()`, `requireApproved()` oder `requireAdmin()` aus `utils/supabase/auth-helpers.ts` verwenden вҖ” diese werfen bei VerstoГҹ einen Error.


Die RLS-Policies in Supabase sind **zusГӨtzliche Absicherung** вҖ” nicht der einzige Schutz. `restaurants`/`comments`-SELECT und `comments`-INSERT sind in RLS ebenfalls an `is_approved()` gebunden.

### Session-Refresh + Zugriffs-Gate (Proxy)

`proxy.ts` (Next.js-16-Nachfolger von `middleware.ts`) ruft `updateSession()` auf jeder Route auf вҖ” hГӨlt Supabase-JWT frisch **und** entscheidet Гјber Redirects (`/login` bzw. `/pending`, s.o.). LГӨuft **nicht** auf `_next/static`, `_next/image` und Bilddateien. Proxy lГӨuft standardmГӨГҹig im Node.js-Runtime (nicht mehr Edge).

### HTTP-Security-Header (`next.config.ts`)

`headers()` in `next.config.ts` setzt app-weit `X-Frame-Options: DENY`, `X-Content-Type-Options: nosniff`, `Referrer-Policy: strict-origin-when-cross-origin` und ein restriktives `Permissions-Policy` (`camera=(), microphone=(), geolocation=(self), usb=()` вҖ” Kamera/Mikrofon/USB bleiben komplett gesperrt, da nirgends genutzt; Geolocation ist seit dem вҖһStandort verwenden"-Button in `LocationSearch.tsx` fГјr die eigene Origin erlaubt, war vorher ebenfalls `()`. Ohne `(self)` schlГӨgt `navigator.geolocation.getCurrentPosition()` mit einem stillen вҖһPermissions policy violation"-Fehler fehl, bevor der Browser Гјberhaupt nach der Nutzer-Erlaubnis fragt). Bewusst **keine** Content-Security-Policy: Google Maps/Places (JS-SDK + Kachel-/Font-Requests) und die Гјber Resend verschickten Auth-Mails laden von genug Drittanbieter-Origins, dass eine CSP erst gegen eine echte Browser-Session aufgebaut/verifiziert werden mГјsste, um nicht versehentlich die Karte oder den Auth-Flow lautlos zu brechen вҖ” offener Folgeschritt statt RГӨtselraten.

### E-Mail-Versand (Resend statt Supabase-SMTP)

Supabase Auth verschickt BestГӨtigungslinks etc. weiterhin selbst (Token-Erzeugung, Verifizierung Гјber den gehosteten `/auth/v1/verify`-Endpunkt bleibt unverГӨndert) вҖ” aber **nicht mehr per eigenem SMTP**, sondern Гјber den [Send Email Hook](https://supabase.com/docs/guides/auth/auth-hooks/send-email-hook):

1. Supabase ruft bei jeder Auth-Mail (`signup`, `recovery`, `magiclink`, `invite`, `email_change`, `reauthentication`) `POST /api/auth/email` auf (`app/api/auth/email/route.ts`) вҖ” signiert nach dem [Standard-Webhooks](https://www.standardwebhooks.com/)-Format.
2. Die Route verifiziert die Signatur mit dem Paket `standardwebhooks` (Secret: `SEND_EMAIL_HOOK_SECRET`, Format `v1,whsec_...`).
3. Aus `token_hash` / `email_action_type` / `redirect_to` wird die gleiche BestГӨtigungs-URL gebaut, die auch Supabases eigene Default-Templates verwenden (`${SUPABASE_URL}/auth/v1/verify?token=...&type=...&redirect_to=...`) вҖ” es gibt **keine eigene Verify-Route**, GoTrue Гјbernimmt PrГјfung + Redirect wie gehabt.
4. Das passende deutsche HTML-Template kommt aus `lib/auth-emails.ts`, verschickt wird Гјber `lib/resend.ts` (`getResendClient()`, Absender `RESEND_FROM_EMAIL`).
5. Bei `reauthentication` gibt es keinen Link, sondern einen 6-stelligen Code (`email_data.token`) zum manuellen Eingeben.

Konfiguration in Supabase Dashboard вҶ’ Authentication вҶ’ Hooks вҶ’ Send Email Hook вҶ’ URL auf `/api/auth/email` zeigen lassen, Secret dort kopieren und als `SEND_EMAIL_HOOK_SECRET` setzen. Ohne konfiguriertes `RESEND_API_KEY`/`SEND_EMAIL_HOOK_SECRET` antwortet die Route mit HTTP 500 (bricht den Auth-Flow bewusst ab, statt E-Mails zu verschlucken).

**BestГӨtigungsscreen nach Signup** (`app/auth/confirm/page.tsx`): `signUp()` in `app/actions/auth.ts` setzt `emailRedirectTo: ${origin}/auth/confirm`. GoTrue verifiziert den Token selbst (hosteter `/auth/v1/verify`-Endpoint) und redirected danach mit einem PKCE-`code` an diese Seite; sie tauscht den Code per `exchangeCodeForSession` gegen eine Session und zeigt Erfolgs- (eingeloggt, Link zur Startseite) oder Fehler-Screen (Link zum Login). **Wichtig:** `${origin}/auth/confirm` muss in Supabase Dashboard вҶ’ Authentication вҶ’ URL Configuration вҶ’ Redirect URLs eingetragen sein, sonst verwirft GoTrue die Redirect-URL still und fГӨllt auf die Site-URL zurГјck.

**Passwort zurГјcksetzen/ГӨndern** (`app/auth/reset-password/page.tsx`): identisches Muster wie oben, aber fГјr den `recovery`-Auth-Mail-Typ. Beide AuslГ¶ser landen auf demselben Ziel:
- **Passwort vergessen** (ausgeloggt): вҖһPasswort vergessen?"-Link im Login-Tab von `LoginForm.tsx` вҶ’ `requestPasswordReset(email)` вҶ’ `supabase.auth.resetPasswordForEmail(email, { redirectTo: `${origin}/auth/reset-password` })`.
- **Passwort ГӨndern** (eingeloggt): вҖһPasswort ГӨndern" im `<UserMenu>`-Kebab-MenГј im Header вҶ’ dieselbe `requestPasswordReset`-Action, aber mit der eigenen Session-E-Mail statt eines Formularfelds вҖ” es gibt bewusst **kein** Formular mit altem/neuem Passwort inline, nur den E-Mail-Link-Weg.
- `requestPasswordReset` meldet in beiden FГӨllen immer Erfolg zurГјck (unabhГӨngig davon, ob die E-Mail existiert oder der Supabase-Call fehlschlГӨgt) вҖ” verhindert, dass sich per Fehlermeldung erraten lГӨsst, welche E-Mails ein Konto haben.
- GoTrue redirected nach Klick auf den Mail-Link mit einem PKCE-`code` an `/auth/reset-password`; die Seite tauscht ihn per `exchangeCodeForSession` gegen eine Session (genau wie `auth/confirm`) und zeigt danach `ResetPasswordForm` (neues Passwort + BestГӨtigung, min. 8 Zeichen) вҖ” das Formular ruft `updatePassword(password)` (`supabase.auth.updateUser({ password })`), was eine aktive Session voraussetzt.
- **Wichtig:** `/auth/reset-password` muss wie `/auth/confirm` in `PUBLIC_PATHS` (`utils/supabase/proxy.ts`) stehen вҖ” vor dem Code-Exchange existiert noch keine Session, ein Gate wГјrde den Link samt `?code=`-Query-Param zu `/login` umleiten.

### `profiles.ts`
- `getPendingProfiles()` вҖ” Admin-only, Liste aller `status = 'pending'`-Accounts

- Immer `cache: "no-store"` вҖ” Г–ffnungszeiten und Foto-URLs ГӨndern sich tГӨglich

### Karten-Dark-Mode
Google ignoriert das `styles`-JSON-Prop **immer**, sobald eine Map-ID gesetzt ist (nГ¶tig fГјr die runden `AdvancedMarker`-Emoji-Pins) вҖ” eine einzelne Map-ID kann sich also nicht per JS auf Light/Dark umstellen. LГ¶sung: zwei Map-IDs, je eine pro Theme, ausgewГӨhlt zur Laufzeit:
- `MapView.tsx` beobachtet `<html data-theme>` per `MutationObserver` (ThemeToggle.tsx mutiert das Attribut direkt ohne Event) + `matchMedia("(prefers-color-scheme: dark)")` fГјr den Systemfall, und wГӨhlt daraus `NEXT_PUBLIC_GOOGLE_MAPS_MAP_ID` (hell) oder `NEXT_PUBLIC_GOOGLE_MAPS_MAP_ID_DARK` (dunkel).
- Die `<Map>`-Komponente bekommt `key={mapId}` вҖ” Google-Map-IDs sind auf einer bestehenden Instanz unverГӨnderlich, daher erzwingt der Key-Wechsel einen sauberen Remount statt eines stillen No-ops.
- Ist `NEXT_PUBLIC_GOOGLE_MAPS_MAP_ID_DARK` nicht gesetzt, fГӨllt Dark Mode auf die helle Map-ID zurГјck (kein Fehler). **Manueller Schritt in der Google Cloud Console nГ¶tig**, um Dark Mode tatsГӨchlich zu sehen: eine zweite Map-ID anlegen und dort einen Kartenstil verknГјpfen, dann `NEXT_PUBLIC_GOOGLE_MAPS_MAP_ID_DARK` setzen.
- Ist gar keine Map-ID gesetzt (weder hell noch dunkel), greift der lokale Fallback: `styles={mapStyle}`/`styles={mapStyleDark}` (`public/map-style.json`/`public/map-style-dark.json`) direkt im Code вҖ” funktioniert dann aber ohne `AdvancedMarker`-Custom-Pins.
- **Zwei unterschiedliche, NICHT austauschbare JSON-Schemas** (per Trial & Error gefunden, 2026-07-16 вҖ” Cloud Console meldete вҖһSyntax-Fehler" beim Versuch, das falsche Schema zu importieren):

  - `public/map-style-cloud-light.json`/`map-style-cloud-dark.json` вҖ” Cloud-based-Maps-Styling-Schema (`{ "variant": "light"|"dark", "styles": [{ "id", "geometry": { "fillColor", "strokeColor", "strokeWidth" }, "label": { "textFillColor", "textStrokeColor", "visible" } }] }`). **Das ist das Format, das die Cloud Console вҶ’ вҖһMap Styles" вҶ’ JSON-Import-Wizard erwartet** вҖ” dort den Inhalt der jeweiligen `map-style-cloud-*.json` einfГјgen, um eine Map-ID mit dem passenden Stil zu verknГјpfen. `strokeWidth` darf hier (anders als evtl. vermutet) durchaus Dezimalwerte haben (z.B. `0.4`) вҖ” der Import-Fehler kam vom falschen Schema, nicht von Nachkommastellen.
  - Bei Г„nderungen an der Optik (Farben, StraГҹenbreiten etc.) **beide** Schema-Paare parallel pflegen, sonst laufen JS-Fallback und Cloud-Console-Stil auseinander.

## Responsive Design & PWA-Ziel

**Die App muss durchgehend sowohl auf Desktop als auch auf Mobile gut aussehen und benutzbar sein** вҖ” nicht nur вҖһfunktioniert technisch", sondern layout- und bedienungsmГӨГҹig auf beiden Formfaktoren geprГјft. - Bei jeder neuen/geГӨnderten Komponente/Seite: Tailwind-Breakpoints (`sm:`/`md:`/`lg:`) bewusst einsetzen, nicht nur die Desktop-Ansicht im Kopf haben. Insbesondere Modals/Slide-Over-Panels (s. `AdminDashboard.tsx`), Tabellen (auf Mobile ggf. horizontal scrollbar oder alternative Kartenansicht) und die Landing-Page-Suche (`HeroSearch`/`FilterBar`/`LocationSearch`) im Blick behalten.
- Wo mГ¶glich per Browser-Test (Playwright) sowohl in einer Desktop- als auch einer Mobile-Viewport-GrГ¶Гҹe verifizieren, bevor ein UI-Task als abgeschlossen gilt (s. Hinweis unter вҖһUI/Frontend-Г„nderungen" in den allgemeinen Arbeitsanweisungen).
- **PWA-Grundausstattung umgesetzt** (`app/manifest.ts`, `app/icon.tsx`/`app/apple-icon.tsx`, `app/icons/*/route.tsx`, `appleWebApp`-Metadata in `app/layout.tsx`, `<InstallPwaInstructions>` auf der Landing-Page) вҖ” noch **kein** Service Worker/Offline-Cache, die App funktioniert installiert also weiterhin nur online. ## Wichtige Muster

- **Server-first**: Datenabruf in Server Components, Mutationen als Server Actions
- **Kein API-Layer**: Keine Route Handlers fГјr CRUD вҖ” direkt Server Actions. Ausnahme: `app/api/auth/email/route.ts`, weil Supabase dies als HTTP-Webhook aufruft (Server Actions sind dafГјr nicht ansprechbar)
- **revalidatePath nach Mutation**: Immer aufrufen um Next.js Cache zu leeren
- **Typen aus `types/database.ts`**: Nie inline tippen вҖ” immer die exportierten Convenience-Typen verwenden (`Restaurant`, `Comment`, `Profile`, `DataSource`, `RestaurantReview`, `RestaurantReviewCategory`, `ReviewWithCategories`, `RestaurantWithComments`, `CommentWithProfile`)

- **Google-APIs immer mit `languageCode=de&regionCode=DE`**: Sowohl serverseitig (REST-Aufrufe in `app/actions/places.ts`, `getPlaceDetails()` **und** `resolvePlaceForImport()`) als auch clientseitig (jedes `<APIProvider>` вҖ” `LocationSearch.tsx`, `MapView.tsx`, `AdminDashboard.tsx` вҖ” trГӨgt `language="de" region="DE"`, wirkt fГјr die ganze darunter geladene Maps-JS-API-Session inkl. - **Spoon-Ratings aus `lib/ratings.ts`**: Emoji/Label nie inline duplizieren вҖ” von dort importieren
- **Farben nie hartkodieren**: Immer CSS-Custom-Properties aus `app/globals.css` verwenden (s. вҖһDark Mode" oben) вҖ” sonst bricht Dark Mode an dieser Stelle
- **Keine Emojis auГҹer Spoon-Rating**: Dekorative Emojis (Status-Icons, Buttons) sind aus der App entfernt вҖ” stattdessen `components/icons.tsx` verwenden. - **Responsive Pflicht**: Jede neue/geГӨnderte UI muss sowohl auf Desktop als auch auf Mobile gut aussehen (s. вҖһResponsive Design & PWA-Ziel" unten) вҖ” kein Feature gilt als fertig, wenn es nur auf einer Breakpoint-GrГ¶Гҹe getestet wurde
